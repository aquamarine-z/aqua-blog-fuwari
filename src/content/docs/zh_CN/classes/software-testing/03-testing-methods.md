# 第三章：软件测试基本方法

## 第1部分：黑盒测试方法

*   **边界值分析法 (BVA)（掌握）**
    *   **原理**：错误极易发生在输入变量边界或其附近。
    *   **取值**：对于区间 $[a, b]$，通常选取 $a-1, a, a+1, b-1, b, b+1$。
    
    > [!IMPORTANT]
    > **📊 四类边界值方法的用例数量计算公式**
    > 
    > 假设系统有 $n$ 个独立输入变量，其用例数计算公式及规则如下：
    > 
    > | 边界值测试方法 | 基础假设 | 核心取点规则（每个变量） | 考试计算公式 | 3个变量时的用例数 ($n=3$) |
    > | :--- | :--- | :--- | :--- | :--- |
    > | **普通边界值测试**<br>(Boundary Value Analysis) | **单缺陷假设**<br>(不考虑边界值同时出现) | $\{\text{min}, \text{min}+, \text{nominal}, \text{max}-, \text{max}\}$<br>每次仅一个变量取边界，其余取正常值 $\text{nominal}$ | $$\mathbf{4n + 1}$$ | $4 \times 3 + 1 = \mathbf{13}$ 个 |
    > | **健壮性测试**<br>(Robustness Testing) | **单缺陷假设**<br>(允许输入越界) | $\{\text{min}-, \text{min}, \text{min}+, \text{nominal}, \text{max}-, \text{max}, \text{max}+\}$<br>包含超出边界的无效值 | $$\mathbf{6n + 1}$$ | $6 \times 3 + 1 = \mathbf{19}$ 个 |
    > | **最坏情况测试**<br>(Worst-Case Testing) | **多缺陷假设**<br>(边界值可同时出现) | $\{\text{min}, \text{min}+, \text{nominal}, \text{max}-, \text{max}\}$<br>对所有变量的 5 个取值进行**全组合** | $$\mathbf{5^n}$$ | $5^3 = \mathbf{125}$ 个 |
    > | **健壮最坏情况测试**<br>(Robust Worst-Case) | **多缺陷假设**<br>(多边界且允许越界) | $\{\text{min}-, \text{min}, \text{min}+, \text{nominal}, \text{max}-, \text{max}, \text{max}+\}$<br>对所有变量的 7 个取值进行**全组合** | $$\mathbf{7^n}$$ | $7^3 = \mathbf{343}$ 个 |
    > 
    > *   **⚠️ 考试避坑指南**：公式中的 $+1$ 代表的是**所有变量都取正常值（$\text{nominal}$）的那 1 个基准用例**。在单缺陷假设下，因为每个变量的 $4$ 或 $6$ 个边界情况是独立测试的，所以最后需要加上这 1 个基准用例。而多缺陷（最坏情况）是笛卡尔积组合，基准用例已包含在幂次组合中，因此不需额外加1。
*   **等价类划分法（掌握）**
    *   **原理**：将输入域划分为若干子集，每个子集内的测试数据被视为等价。分为**有效等价类**和**无效等价类**。
    *   **分类**：
        1.  **弱一般等价类**：只考虑有效值，单缺陷假设。
        2.  **弱健壮等价类**：考虑有效和无效值，单缺陷假设。
        3.  **强一般等价类**：只考虑有效值，多缺陷假设（测试有效等价类的所有组合）。
        4.  **强健壮等价类**：考虑有效和无效值，多缺陷假设（测试所有有效和无效等价类的组合）。
*   **判定表方法（掌握）**
    *   **适用场景**：输入条件多且存在复杂的逻辑组合关系。
    *   **结构**：条件桩（输入属性）、动作桩（输出属性）、条件项（输入赋值如 Y/N 或 1/0）、动作项（对应的输出结果）。每一列代表一条“规则”。
    *   **优化**：如果某些属性取 1 还是 0 不影响输出，可以使用符号 `-` 合并规则。
*   **因果图法（掌握）**
    *   **原理**：通过图形分析输入条件（因）与输出动作（果）之间的逻辑关系，并转化为判定表。
    *   **逻辑关系符号**：蕴含（Identity）、非（NOT）、或（OR $\vee$）、与（AND $\wedge$）。
*   **Pairwise方法与正交试验法（简要了解）**
    *   **Pairwise方法**：基于经验，绝大多数错误是由两个属性的交互组合引起的，因此设计用例覆盖所有“任意两属性值”的组合，可大幅减少组合测试的用例数量。
    *   **正交试验法**：利用正交表从大量的试验点中挑选适量的、有代表性的点安排测试。

## 第2部分：白盒测试方法

### 1. 逻辑覆盖（重点）

*   **概念区分**：
    *   **条件 (Condition)**：不可再分的简单逻辑表达式（如 `a > 0`）。
    *   **判定 (Decision)**：包含一个或多个条件的复合逻辑表达式（如整个 `if (a > 0 && b < 10)`）。
    
    > [!TIP]
    > **💡 极简生活例子：景区门票优惠（判定 vs. 条件）**
    > 
    > 景区规定：**如果是老人（条件1） 并且 持有优待证（条件2），就可以免票（判定）。**
    > *   **条件**：条件1：是否是老人（真/假）；条件2：是否有优待证（真/假）。
    > *   **判定**：最终结果：是否免票（真/假）。
    > 
    > **🚀 考试常考核心：满足“判定覆盖”但不满足“条件覆盖”**
    > 
    > 假设判定表达式为 `if (A || B)`：
    > *   **测试用例组**：
    >     *   用例1：`A = 真, B = 假` $\rightarrow$ 判定结果为 **真**
    >     *   用例2：`A = 假, B = 假` $\rightarrow$ 判定结果为 **假**
    > *   **为什么满足判定覆盖？** 因为判定的最终结果“真”（用例1）和“假”（用例2）都跑过一遍了。
    > *   **为什么不满足条件覆盖？** 对于单个条件 `B`，在用例1中是“假”，在用例2中也是“假”，**`B = 真` 的情况完全没有被测试到**。这就是为什么“满足判定覆盖不一定能满足条件覆盖”。

*   **覆盖指标**：
    1.  **语句覆盖 / 节点覆盖**：程序中的每个可执行语句（CFG 中的节点）至少执行一次。
    2.  **判定覆盖 / 分支覆盖 / 边覆盖**：每个判定语句的“真”和“假”分支（CFG 中的每条边）至少执行一次。
    3.  **条件覆盖**：每个判定语句中的每个具体条件都取得过“真”和“假”结果。
    4.  **条件/判定覆盖 (C/DC)**：同时满足判定覆盖和条件覆盖。
    5.  **条件组合覆盖**：每个判定中所有具体条件取值的可能组合至少出现一次。
    6.  **修改的条件/判定覆盖 (MC/DC)**：在满足 C/DC 的基础上，要求在每个判定中，每个条件 C 都曾独立地影响判定的最终输出（即当其他条件不变，仅改变 C 的值时，判定的结果随之改变）。
    
    > [!TIP]
    > **💡 极简生活例子：一票否决权（MC/DC）**
    > 
    > MC/DC 的核心思想是：**证明每一个孤立的“条件”，都能够独立地改变整个“判定”的最终结果**。
    > 假设判定为：`if (评委A && (评委B || 评委C))`。我们要证明“评委A”的投票是有决定性意义的（具有独立改变结果的能力）：
    > *   **用例1**：`A = 1 (赞成), B = 1 (赞成), C = 0 (反对)` $\rightarrow$ 最终结果为 **1 (通过)**
    > *   **用例2**：`A = 0 (反对), B = 1 (赞成), C = 0 (反对)` $\rightarrow$ 最终结果为 **0 (否决)**
    > *   **对比用例1和2**：评委 B 和 C 的投票**完全没有变**（都是 `B = 1, C = 0`），仅仅因为评委 A 从赞成（1）变成了反对（0），最终结果就从通过（1）变成了否决（0）。这就证明了 **A 具有独立改变结果的能力**。这组用例就是 A 满足 MC/DC 的用例对。
    
    7.  **路径覆盖**：覆盖程序中所有可能的执行路径。
    
    > [!IMPORTANT]
    > **📊 白盒逻辑覆盖标准的强弱包含关系**
    > 
    > 根据蕴含关系（即“满足 A 覆盖则一定自动满足 B 覆盖”），它们的强弱级联网络结构如下：
    > *   **链条一（多缺陷条件链）**：条件组合覆盖 $\implies$ MC/DC $\implies$ 判定/条件覆盖 (C/DC) $\implies$ 判定覆盖 $\implies$ 语句覆盖
    > *   **链条二（路径控制链）**：路径覆盖 $\implies$ 判定覆盖 $\implies$ 语句覆盖
    > *   **🚨 必考警示：条件覆盖与判定覆盖的非蕴含关系**：
    >     *   **判定覆盖 $\not\implies$ 条件覆盖**：跑遍了真假分支，但判定内部的某个子条件可能从未取到过真（或假）。
    >     *   **条件覆盖 $\not\implies$ 判定覆盖**：每个子条件的真假都测到了，但由于它们互相抵消，最终判定输出可能全为真（或全为假），漏掉了真假分支。
    >     *   **MC/DC 的核心意义**：对于 $N$ 个条件，条件组合覆盖需要 $2^N$ 级用例，而 MC/DC 只需要 $N+1$ 个用例，是用最小代价换取安全保障的业界黄金标准。

### 2. 基本路径测试（重点）

*   **原理**：由于路径覆盖在存在循环时用例数呈指数级增加，因此退而求其次，测试一组线性无关的“基本路径基”。
*   **控制流图（CFG）与 DD路径图（Decision-to-Decision Path）**：
    *   **DD路径**本质上就是**代码控制流图的“压缩合并”**。
    
    > [!TIP]
    > **💡 极简比喻：地铁运行与大站快车**
    > *   **普通流图**：把地铁线上的每一个小站（每一行代码）都画成一个独立的节点。
    > *   **DD路径图**：把两个换乘大站（决策点/分支点）之间的所有普通小站，合并压缩成一根线。
    > *   **目的**：压缩无分支的串行节点，只保留关键的决策点，减少白盒测试分析路径时的复杂度。

*   **圈复杂度 (Cyclomatic Complexity)**：度量了基本路径基中的路径数目。
    *   **计算公式（三种等价方法，建议双重校验）**：
        1.  **边点公式（非强连通图，单入口单出口）**：
            $$V(G) = e - n + 2$$
            *(其中 $e$ 为控制流图的边数，$n$ 为节点数。)*
        2.  **区域判定法**：
            $$V(G) = \text{控制流图在平面上划分的区域数} = \text{封闭区域数} + 1$$
        3.  **判定节点快速公式（★ 考试提速神器）**：
            $$V(G) = P + 1$$
            *(其中 $P$ 为控制流图中的**判定节点（Predicate Nodes）**数，即出度 $\ge 2$ 的分支节点数。如简单 `if`、`while`、`for`。如果条件中出现复合逻辑（如 `&&`、`||`），控制流图会将其拆分为多个单条件的分支节点，应以**画出来的控制流图中的分支节点数**为准。)*

### 3. 数据流测试（重点）

*   **基本概念**：
    *   **定义 (def)**：变量被写操作（如赋值、初始化 `x = 5`）的节点。
    *   **使用 (use)**：变量值被读操作的节点。分为：
        *   **c-use** (Computation use，计算使用)：读该变量是为了**进行数值计算或赋值**（如 `y = x + 1`）。
        *   **p-use** (Predicate use，谓词使用)：读该变量是为了**进行逻辑判断、控制程序分支**（如 `if (x > 0)`）。
    *   **定义-清除路径 (def-clear path)**：起点为该变量的定义节点，终点为该变量的使用节点，且中途没有任何节点对该变量进行过重新定义。

    > [!TIP]
    > **💡 极简比喻：交接接力棒**
    > *   你在 A 点把接力棒（变量 `x`）交给了小明（定义 `x = 5`）。
    > *   小明一路跑到了 B 点，中途**没有任何人替换**这根棒子（这就是**定义-清除路径**）。
    > *   到了 B 点：
    >     *   如果是把接力棒拿去称重或测量长度（进行计算使用，即 `c-use`）。
    >     *   如果是根据接力棒的颜色决定往左跑还是往右跑（做谓词条件分支判断，即 `p-use`）。

*   **覆盖指标**：
    *   **全定义 (All-Defs)**：对每个变量的每个定义节点，覆盖到至少一个使用节点的定义-清除路径。
    *   **全使用 (All-Uses)**：对每个变量的每个定义节点和对应的每个使用节点，都覆盖一条定义-清除路径。
    *   **全定义-使用路径 (All-DU-Paths)**：覆盖所有可能的定义-清除路径（允许存在简单循环）。

## 第3部分：其他测试方法

*   **变异测试（简要了解）**：通过往代码中注入微小错误（变异体 Mutant，如将 `+` 替换为 `-`），评估已有测试用例是否足够有效。如果测试用例能输出不同的结果，则称“击杀（Killed）”了变异体。
*   **基于约束的测试（简要了解）**：将程序中的路径可达条件转换为数学/逻辑约束公式 (Constraint)，利用逻辑求解器（如 Z3 SMT 求解器）进行自动求解，自动生成能到达特定目标点的输入数据。
