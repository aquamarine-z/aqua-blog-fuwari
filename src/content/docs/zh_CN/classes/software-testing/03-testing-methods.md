# 第三章：软件测试基本方法

## 第1部分：黑盒测试方法

*   **边界值分析法 (BVA)（掌握）**
    *   **原理**：错误极易发生在输入变量边界或其附近。
    *   **取值**：对于区间 $[a, b]$，选取 $a-1, a, a+1, b-1, b, b+1$。
    *   **分类**：
        1.  **普通边界值测试**：单缺陷假设（一次只让一个变量取极值），只考虑有效值。
        2.  **健壮性测试**：单缺陷假设，允许变量取无效值（边界外的 $a-1, b+1$）。
        3.  **最坏情况测试**：多缺陷假设（多个变量同时取极值），只考虑有效值。
        4.  **健壮最坏情况测试**：多缺陷假设，允许变量取无效值。
*   **等价类划分法（掌握）**
    *   **原理**：将输入域划分为若干子集，每个子集内的测试数据被视为等价。分为**有效等价类**和**无效等价类**。
    *   **分类**：
        1.  **弱一般等价类**：只考虑有效值，单缺陷假设。
        2.  **弱健壮等价类**：考虑有效和无效值，单缺陷假设。
        3.  **强一般等价类**：只考虑有效值，多缺陷假设（测试有效等价类的所有组合）。
        4.  **强健壮等价类**：考虑有效和无效值，多缺陷假设（测试所有有效和无效等价类的组合）。
*   **判定表方法（掌握）**
    *   **适用场景**：输入条件多且存在复杂的逻辑组合关系。
    *   **结构**：条件桩（输入属性）、动作桩（输出属性）、条件项（输入赋值如 Y/N 或 1/0）、动作项（对应的输出结果）。每一列代表一条“规则”。
    *   **优化**：如果某些属性取 1 还是 0 不影响输出，可以使用符号 `-` 合并规则。
*   **因果图法（掌握）**
    *   **原理**：通过图形分析输入条件（因）与输出动作（果）之间的逻辑关系，并转化为判定表。
    *   **逻辑关系符号**：蕴含（Identity）、非（NOT）、或（OR $\vee$）、与（AND $\wedge$）。
*   **Pairwise方法与正交试验法（简要了解）**
    *   **Pairwise方法**：基于经验，绝大多数错误是由两个属性的交互组合引起的，因此设计用例覆盖所有“任意两属性值”的组合，可大幅减少组合测试的用例数量。
    *   **正交试验法**：利用正交表从大量的试验点中挑选适量的、有代表性的点安排测试。

## 第2部分：白盒测试方法

### 1. 逻辑覆盖（重点）

*   **概念区分**：
    *   **条件**：不可再分的简单逻辑表达式（如 `a > 0`）。
    *   **判定**：包含一个或多个条件的复合逻辑表达式（如 `if (a > 0 && b < 10)`）。
*   **覆盖指标**：
    1.  **语句覆盖 / 节点覆盖**：程序中的每个可执行语句（CFG 中的节点）至少执行一次。
    2.  **判定覆盖 / 分支覆盖 / 边覆盖**：每个判定语句的“真”和“假”分支（CFG 中的每条边）至少执行一次。
    3.  **条件覆盖**：每个判定语句中的每个具体条件都取得过“真”和“假”结果。
    4.  **条件/判定覆盖 (C/DC)**：同时满足判定覆盖和条件覆盖。
    5.  **条件组合覆盖**：每个判定中所有具体条件取值的可能组合至少出现一次。
    6.  **修改的条件/判定覆盖 (MC/DC)**：在满足 C/DC 的基础上，要求在每个判定中，每个条件 C 都曾独立地影响判定的最终输出（即当其他条件不变，仅改变 C 的值时，判定的结果随之改变）。
    7.  **路径覆盖**：覆盖程序中所有可能的执行路径。

---

### 2. 基本路径测试（重点）

*   **原理**：由于路径覆盖在存在循环时用例数呈指数级增加，因此退而求其次，测试一组线性无关的“基本路径基”。
*   **圈复杂度 (Cyclomatic Complexity)**：度量了基本路径基中的路径数目。
    *   若图为单入口单出口（非强连通图），圈复杂度 $V(G) = e - n + 2$（$e$ 为边数，$n$ 为节点数）。
    *   若图为强连通图，圈复杂度 $V(G) = e - n + 1$。

---

### 3. 数据流测试（重点）

*   **基本概念**：
    *   **定义 (def)**：变量被赋值或初始化的节点。
    *   **使用 (use)**：变量值被读取的节点。分为 **c-use**（在计算/赋值中使用）和 **p-use**（在分支条件谓词中使用）。
    *   **定义-清除路径 (def-clear path)**：从定义点到使用点的路径，且路径上其他节点没有对该变量进行重新定义。
*   **覆盖指标**：
    *   **全定义 (All-Defs)**：对每个变量的每个定义节点，覆盖到至少一个使用节点的定义-清除路径。
    *   **全使用 (All-Uses)**：对每个变量的每个定义节点和对应的每个使用节点，都覆盖一条定义-清除路径。
    *   **全定义-使用路径 (All-DU-Paths)**：覆盖所有可能的定义-清除路径（允许存在简单循环）。

## 第3部分：其他测试方法

*   **变异测试（简要了解）**：通过往代码中注入微小错误（变异体 Mutant，如将 `+` 替换为 `-`），评估已有测试用例是否足够有效。如果测试用例能输出不同的结果，则称“击杀（Killed）”了变异体。
*   **基于约束的测试（简要了解）**：将程序中的路径可达条件转换为数学/逻辑约束公式 (Constraint)，利用逻辑求解器（如 Z3 SMT 求解器）进行自动求解，自动生成能到达特定目标点的输入数据。
