# 第五章：单元测试与集成测试

### 1. 静态代码审查

*   **代码走查 (walkthrough)**：非正式会议，由代码作者讲解，开发人员为主。
*   **会议审查 (inspection)**：正式会议，项目组全员（包括测试）参加，严格按缺陷检查表逐行检查，记录缺陷生成报告。

---

### 2. 单元测试基本概念

*   **驱动程序 (Driver)**：模拟被测模块的上级模块，负责调用被测单元，向其传送测试数据。
*   **桩程序 (Stub)**：模拟被测模块调用的下级模块，用于接收被测单元的调用并返回固定值。

---

### 3. Mock、Stub 与 Fake 区别

*   **Stub**：提供固定的、预定义的数据响应调用，通常用于占位，不涉及行为验证。
*   **Mock**：比 Stub 复杂，具有**行为验证/断言**功能，可检查特定接口是否被调用、调用次数及顺序。
*   **Fake**：有实际的、简化的工作实现（如内存数据库 H2 替代真实数据库），不强调断言的存在。
*   **常用工具**：JUnit（Java 单元测试业界标准），Mockito（Mock 框架），JaCoCo（代码覆盖率统计工具）。

---

### 4. 集成测试模式

*   **非渐增式（大棒集成 Big Bang）**：每个模块先各自进行单元测试，然后把所有模块一次性混合集成。**难以定位错误位置**，只适合小系统。
*   **渐增式**：
    *   *自顶向下 (Top-down)*：从主控模块开始向下集成。优点是较早发现架构层、高层逻辑问题；缺点是底层模块测试晚，**需要写大量的桩**。
    *   *自底向上 (Bottom-up)*：从最底层模块向上集成。优点是底层模块测试充分；缺点是高层控制模块测试晚，**需要写大量的驱动**。
    *   *三明治式 (Sandwich)*：混合策略。高层采用自顶向下，底层采用自底向上，在中间层汇合。
