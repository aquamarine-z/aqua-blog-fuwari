# 第五章：单元测试与集成测试

### 1. 静态代码审查

*   **代码走查 (walkthrough)**：非正式会议，由代码作者讲解，开发人员为主。
*   **会议审查 (inspection)**：正式会议，项目组全员（包括测试）参加，严格按缺陷检查表逐行检查，记录缺陷生成报告。

---

### 2. 单元测试基本概念

*   **驱动程序 (Driver)**：模拟被测模块的上级模块，负责调用被测单元，向其传送测试数据。
*   **桩程序 (Stub)**：模拟被测模块调用的下级模块，用于接收被测单元的调用并返回固定值。

---

### 3. Mock、Stub 与 Fake 区别

在写单元测试或接口测试时，隔离外部依赖（数据库、网络接口、第三方服务）通常使用这三种模拟技术：

| 角色 | 通俗比喻 | 特点与工作原理 |
| :--- | :--- | :--- |
| **Stub (桩对象)** | **冷酷的自动回复机** | **只管提供数据响应**。不管外界如何调用，都只返回预设好的死数据，用以顶替未实现的依赖使测试继续走下去，自身不包含行为验证断言。<br />*例子*：测试付款模块，下游的第三方支付尚未就绪，Stub 永远返回 `支付成功: true`。 |
| **Mock (模拟对象)** | **带密录设备的间谍** | **不仅提供响应，还要暗中监视与核对**。它会忠实记录被测单元对它进行了几次调用、传入了什么参数、调用顺序是否正确。在测试结束时需要进行行为**断言验证 (Verify)**。<br />*例子*：测试发送邮件，测试用例会验证：“测试结束时，MockMailServer 确实被调用并发送了 1 封邮件吗？” |
| **Fake (伪对象)** | **简易的高仿玩具** | **它是一个真实可用的简化版实现**。由于真实环境下的组件太重（如重量级商业数据库）、耗时太长或有网络限制，因而采用极轻量的单机/内存版替换。<br />*例子*：由于真正的企业 Oracle 数据库配置太重，单机测试时使用一个轻量级的内存数据库（如 SQLite 或 H2）来跑测试。 |

*   **常用工具**：JUnit（Java 单元测试业界标准），Mockito（Mock 框架），JaCoCo（代码覆盖率统计工具）。

---

### 4. 集成测试模式

*   **非渐增式（大棒集成 Big Bang）**：每个模块先各自进行单元测试，然后把所有模块一次性混合集成。**难以定位错误位置**，只适合小系统。
*   **渐增式**：
    *   *自顶向下 (Top-down)*：从主控模块开始向下集成。优点是较早发现架构层、高层逻辑问题；缺点是底层模块测试晚，**需要写大量的桩**。
    *   *自底向上 (Bottom-up)*：从最底层模块向上集成。优点是底层模块测试充分；缺点是高层控制模块测试晚，**需要写大量的驱动**。
    *   *三明治式 (Sandwich)*：混合策略。高层采用自顶向下，底层采用自底向上，在中间层汇合。

---

### 5. 契约测试 (Contract Testing / CDC)

在分布式微服务架构中，多个服务接口的联调与集成测试成本高昂。**契约测试（多采用消费者驱动，Consumer-Driven Contract Testing, CDC）**提供了一种轻量级、去中心化的接口测试方案：

*   **核心思想**：服务的消费者（前端/下游服务）与服务提供者（后端/上游服务）签署一份“法律契约 (Contract)”。
*   **工作机制 (以“点外卖”为例)**：
    *   **消费者（前端/Consumer）** 签署契约：“只要我给你发送请求 `{"item": "Bacon Burger"}`，你（后端）就**必须**返回 `{"price": 15, "status": "delivered"}`。”
    *   **前端测试**：前端拿着这份契约单机运行，Mock 服务根据契约完美扮演后端角色，前端不用等后端写完即可独立完成前端交互的全部测试。
    *   **后端测试**：后端在自己的持续集成（CI）流水线中，自动读取这份契约，向真实的后端接口发送定义的请求，验证接口吐出的字段格式是否跟契约**一模一样**。
*   **优势**：前后端完全解耦，甚至不需要真正建立网络连接，只要各自独立跑通并符合该“契约”，即可保证在真实环境中联调万无一失。
