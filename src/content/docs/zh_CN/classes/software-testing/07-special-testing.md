# 第七章：专项测试

## 1. 性能测试 (Performance Testing)

*   **核心性能指标**：并发用户数、吞吐量（RPS/TPS）、系统响应时间。
*   **计算公式**：
    $$RPS + SC + \text{Thinking Time} = \text{Concurrent Users}$$
    *(其中 $SC$ 为并发连接数，$\text{Thinking Time}$ 为用户思考时间。)*
*   **性能测试类型划分**：
    *   **基准测试 (Benchmark)**：在系统单用户或无压力环境下运行，获取基础性能指标基线。
    *   **负载测试 (Load Testing)**：逐步增加系统负载，测试系统在满足特定 SLA 指标下的最大工作承载能力。
    *   **压力测试 (Stress Testing)**：模拟超负荷和极端流量条件，将系统压至崩溃状态，检测其恢复能力与最大吞吐瓶颈。
    *   **渗入测试 (Soak Testing)**：长时间在高负载下运行系统（如 24 小时以上），检测系统是否存在内存泄露、GC 回收延迟或系统资源枯竭。
    *   **全链路压测**：基于线上真实环境、真实业务规模，采用流量染色、数据隔离等技术开展的线上大规模多系统协同压力测试（如阿里开源的 Takin）。
*   **常用工具**：Apache JMeter（开源的多线程分布式压力测试工具）。

---

## 2. 安全性测试 (Security Testing)

*   **安全三大目标 (CIA)**：保密性 (Confidentiality)、完整性 (Integrity)、可用性 (Availability)。
*   **安全威胁模型 (STRIDE)**：
    *   **S**poofing（假冒身份）
    *   **T**ampering（篡改数据）
    *   **R**epudiation（否认行为）
    *   **I**nformation Disclosure（信息泄露）
    *   **D**enial of Service（拒绝服务）
    *   **E**levation of Privilege（特权提升）
*   **经典 Web 攻击与测试重点**：
    *   **SQL 注入**：输入特殊字符拼接成永真逻辑（如 `OR '1'='1`），绕过数据库校验逻辑。
    *   **XSS (跨站脚本攻击)**：往页面表单注入恶意 JavaScript 脚本，在用户浏览器执行盗取 Cookie。
    *   **垂直/水平越权**：直接修改 URL 或参数中的敏感对象 ID（如 `acct=6065` 越权访问他人账户）。
*   **特殊安全检测手段**：
    *   **Fuzz Testing (模糊测试)**：往程序自动输入大量无规则、畸形的半随机数据，检测程序是否会因为未捕获异常而崩溃（Crash）。
    *   **渗透测试 (Penetration)**：模拟黑客行为，使用各类专项探测工具对目标系统进行深度漏洞利用测试。
*   **常用工具**：Burp Suite、OWASP ZAP、W3af。

---

## 3. 易用性测试：A/B 测试 (A/B Testing)

*   **基本原理**：在线上真实流量中划分不同比例的用户桶，分别运行 A（基线版）和 B（新策略版）两个版本，收集真实转化率、留存率数据以决策产品优化方向。
*   **流量分流机制**：使用一致性哈希算法（如 **MurmurHash**）对用户唯一标识（如 UUID）进行哈希分桶，保证分流均匀和粘性。通过 OpenResty 等网关组件支持多层流量正交与互斥机制，使有限的流量可以支持多个不相关实验并行开展。

---

## 4. 混沌工程 (Chaos Engineering)

*   **核心理念**：通过主动向生产环境注入分布式故障（如随机杀死容器、模拟网关高延迟等），验证系统的韧性、监控报警度与服务降级容错机制（“以毒攻毒”）。
*   **混沌设计原则**：围绕系统稳定状态行为建立假设、模拟多样化的真实世界事件、直接在生产环境中运行、持续自动化运行、最大程度控制故障的爆炸半径。
*   **常用工具**：ChaosBlade（阿里开源）、Chaos Mesh（面向 Kubernetes 的原生混沌测试平台）。
