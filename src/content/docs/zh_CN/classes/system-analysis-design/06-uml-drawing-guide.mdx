# 🎨 UML 画图专项教程与标准例图库

本教程整合了《系统分析与设计》课程中涉及的所有核心图表的**图形元素含义、关系箭头标准以及期末大题标准解答图例**，并统一使用高质量的 Mermaid 语法进行重绘与解析。

---

## 一、 用例图（Use Case Diagrams）

用例图用于捕获系统的静态功能范围，展现外部参与者与系统用例之间的交互关系。

### 1. 用例图核心元素含义说明

*   **系统边界（System Boundary）**：用**矩形**表示，代表系统的内部与外部的隔离界线。系统的所有用例必须画在边界内，外部参与者必须画在边界外。
*   **参与者（Actor）**：用**火柴人（或小人符号）**表示，代表在系统边界之外与系统进行交互的外部实体（可以是人、外部硬件设备，或者其他第三方系统）。
*   **用例（Use Case）**：用**椭圆**表示，代表系统能够为参与者提供的一个完整且具有独立业务价值的功能或交互流。

### 2. 用例关系与 Mermaid 规范

用例之间主要存在三种核心关联关系：

*   **包含（`<<include>>`）**：**虚线箭头，从主用例指向包含用例**。表示主用例在执行时，包含用例**必须执行**。通常用于提取多个用例的公共子步骤。
*   **扩展（`<<extend>>`）**：**虚线箭头，从扩展用例反向指向主用例**。表示在满足特定条件（扩展点）时，可选地触发扩展用例。主用例的执行不依赖于扩展用例。
*   **泛化（Generalization）**：**实线带空心三角箭头，从子用例指向父用例**。表示子用例是父用例的一种特化具体实现，承接父用例的结构和行为。

<Mermaid code={`

    subgraph UseCaseRelations [用例关系示意图]
        direction LR
        Main(["主用例"]) -. "«include»" .-> Include(["包含用例<br>(必须执行的子步骤)"])
        Extend(["扩展用例<br>(在特定条件下触发)"]) -. "«extend»" .-> Main
        Child(["子用例<br>(特化具体实现)"]) -- "泛化关系" --> Parent(["父用例"])
    end

## 二、 系统顺序图（SSD）与对象交互序列图（Sequence Diagrams）

用于描述系统交互在时间维度上的动态协作过程。分析阶段使用**系统顺序图（SSD）**，设计阶段使用**对象交互序列图**。

### 1. 序列图核心元素含义说明

*   **参与者（Actor）**：位于图的最左侧，发起业务请求的外部用户或系统角色。
*   **生命线（Lifeline）**：顶部为矩形框，格式为 `对象名:类名`（若省略对象名，如 `:类名` 则表示匿名对象）。下方延伸的**垂直虚线**代表对象在系统运行中的生命周期存活时间。
*   **激活期/控制焦点（Activation Bar / Focus of Control）**：生命线虚线上的**垂直窄矩形条**。代表该对象当前正在占用 CPU 时间片，执行某个操作或方法。
*   **组合碎片（Combined Fragments）**：带标签的框图，用于控制消息流的执行逻辑：
    *   `alt`：条件分支（相当于代码中的 `if-else`）。
    *   `loop`：循环执行（相当于代码中的 `for/while` 循环，框左上角需标明循环守卫条件）。
    *   `opt`：可选执行（相当于无 `else` 的单分支 `if`）。

### 2. 消息线（Messages）与箭头规范

*   **同步调用（Synchronous Call）**：**实线 + 实心三角箭头（`->>`）**。调用方发送消息后挂起，必须等待接收方处理完毕并返回结果。
*   **返回消息（Return Message）**：**虚线 + 开放式箭头（`-->>`）**。代表某个同步方法执行完毕，将返回值传回给调用者。
*   **异步调用（Asynchronous Call）**：**实线 + 开放式箭头**。调用方发出消息后立即继续执行，不等待接收方响应。
*   **自调用（Self-Call）**：折线指向本对象的另一个激活条。代表对象内部的方法调用（如持久层类内部进行数据插入）。

## 三、 类图与设计类图（Class Diagrams / DCD）

类图用于描述系统的静态结构。概念类图（CCD）侧重于现实概念，设计类图（DCD）面向技术编码实现。

### 1. 类图核心元素含义说明

#### 1.1 类的三段式表示
类在图中用矩形框表示，从上到下分为三格：
1.  **第一格：类名**。接口用 `<<interface>>` 双角括号标识，抽象类名用*斜体*表示。
2.  **第二格：属性（Attributes）**。格式为：`可见性 属性名 : 数据类型 = 默认值`。
3.  **第三格：方法（Methods）**。格式为：`可见性 方法名(参数名 : 参数类型) : 返回值类型`。

#### 1.2 可见性修饰符（Visibility）
*   `+`：**Public（公共）**。外部任意类均可直接访问和调用。
*   `-`：**Private（私有）**。仅在本类内部可见，子类与外部均无法访问（DCD 的属性通常全部设为私有）。
*   `#`：**Protected（受保护）**。仅在本类及子类（继承体系）内部可见。

### 2. 六大核心关系与箭头标准

UML 类图的关系强弱顺序为：**泛化 = 实现 > 组合 > 聚合 > 关联 > 依赖**。

| 关系类型 | 线型 | 箭头形状 | 语义说明与代码实现特征 | Mermaid 语法 |
| :--- | :--- | :--- | :--- | :--- |
| **泛化（继承）** | 实线 | 空心三角形 | **子类指向父类**。表示子类继承父类的结构与行为。代码实现：`extends`。 | `SubClass --\|> SuperClass` |
| **实现（Realization）** | 虚线 | 空心三角形 | **实现类指向接口**。表示类实现抽象接口中声明的方法。代码实现：`implements`。 | `ImplClass ..\|> Interface` |
| **组合（Composition）** | 实线 | 实心黑色菱形 | **从部分指向整体（菱形在整体端）**。强“整体-部分”关系。部分与整体生死与共，部分不能脱离整体独立存在。代码实现：在 `Order` 实例化时直接 `new` 订单项。 | `Whole *-- Part` |
| **聚合（Aggregation）** | 实线 | 空心菱形 | **从部分指向整体（菱形在整体端）**。“整体-部分”关系，但耦合度较弱，部分可以脱离整体而单独存在（如：部门与员工）。代码实现：外部传入参数赋值给成员变量。 | `Whole o-- Part` |
| **关联（Association）** | 实线 | 开放实心箭头 | **一类指向另一类**。类与类之间有结构化的、长期的连接关系。表示类 A 将类 B 的实例作为自己的全局成员属性。通常需标明多重度（如 `1`, `0..*`）。 | `ClassA --> ClassB` |
| **依赖（Dependency）** | 虚线 | 开放实心箭头 | **一类指向另一类**。临时、弱使用关系。A 类的方法在运行期临时使用了 B 类的实例（如：B 作为 A 方法的入参、局部变量，或者 A 方法内静态调用了 B）。 | `ClassA ..> ClassB` |

## 四、 状态图（State Machine Diagrams）

状态图用于刻画一个单一业务对象在其生命周期内，由于接收外部事件（Event）而发生的状态变迁（State Transition）。

### 1. 状态图核心元素含义说明

*   **初态（Initial State）**：用**黑色实心圆点**表示，代表状态机生命周期的起点，一个状态图只能有一个初态。
*   **终态（Final State）**：用**双圆环（同心圆内带实心圆点）**表示，代表对象生命周期的终结，可以有多个终态或没有终态。
*   **状态（State）**：用**圆角矩形**表示，写有该状态的名称（通常为英文大写）。代表对象在生命周期中某一时间段所处的一种稳定物理或逻辑状态。
*   **状态迁移线（Transition）**：**带箭头的实线**。代表状态发生改变的路径。
    *   **标符格式**：`事件 [守卫条件] / 动作`（Event [Guard] / Action）。
    *   **事件**：触发迁移的外部输入（如“提交审核”）。
    *   **守卫条件**：布尔表达式，只有条件为真时迁移才会被触发（如 `[信用分 > 700]`）。
    *   **动作**：迁移发生时系统执行的瞬时行为，执行完后立即进入新状态。

## 五、 活动图（Activity Diagrams）

活动图用于描述系统的工作流或算法逻辑流程，侧重于动作之间的控制流，类似于高级流程图。

### 1. 活动图核心元素含义说明

*   **起点与终点**：与状态图相同，用黑色实心圆点代表流程起点，双圆环代表流程终点。
*   **动作状态（Action / Activity）**：用**圆角矩形**表示，代表流程中需要执行的具体操作或步骤（如“发起HTTP请求”）。
*   **控制流（Control Flow）**：**带箭头的实线**，用于指出动作执行的前后时序关系。
*   **决策点与合并点（Decision / Merge）**：用**空心菱形**表示。
    *   **决策分流**：一条输入流进入，根据不同的**判定条件（用中括号 `[条件]` 标明守卫）**流出多条不同分支。
    *   **合并合流**：多条分支汇合进入，引出一条单一输出流向下传递。

### 2. 实战例图：新闻网页抓取算法控制流

<Mermaid code={`

### 2. Spring Cloud 微服务组件部署拓扑图

展示微服务分布式环境下的高可用组织形式。

*   **API网关（API Gateway）**：外部请求的唯一入口，负责路由分发、认证鉴权与流量拦截控制。
*   **微服务（Microservice）**：高度自治、围绕独立业务领域构建的细粒度后端服务。
*   **服务注册中心（Registry）**：保存所有可用服务实例的名册，管理实例心跳与服务注册发现。
*   **独占数据库（Database）**：各微服务之间实行数据物理隔离，禁止跨库直接 `JOIN`。
*   **消息队列（MQ）**：通过发布订阅机制实现微服务之间的异步解耦与削峰填谷。
